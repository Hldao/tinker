## 一、一句话定位

> **Tinker / 捣鼓 = 给"用 AI 创造但不必懂代码"的人的工作室。**

**反向定位**：
- GitHub 是给写代码的人的工作室
- 我们是给"捣鼓东西的人"的工作室 —— 他们用 AI，做出真的能跑的东西，但**不必是程序员**

### 不是什么（核心反向定位）

| 不是 | 区别 |
|---|---|
| 论坛 | 论坛是"信息单元的集市"，我们是"创作过程的现场" |
| GitHub 山寨 | 不要 repo 也行，欢迎 v0/Bolt/Lovable 链接 |
| ProductHunt | 不是"上线导向"，"在做"也是主舞台 |
| IndieHackers | 不是程序员中心氛围 |
| 技能学习平台 | 不教程，靠看真实工作现场学习 |

### 类比坐标

| 像谁 | 哪里不一样 |
|---|---|
| GitHub | 不要 repo 也行 · 反程序员气质 |
| Notion 公开页 | 加上"被看见"的社交层 |
| Linear | 但是公开的 · 不是公司内工作流 |
| ProductHunt | 主舞台给"在做"，不只给"上线" |

## 二、产品哲学

### 8 条核心哲学（继承自 petfinder · 完整原文见 `petfinder/docs/06-product-philosophy.md`）

1. **不挡用户的路**
2. **反冒领的真实边界**（本产品语境：**反"AI 装大佬"**）
3. **关键节点必须在平台内**
4. ~~WeChat 合规底线~~（本产品不强相关，但海外/国内都遵守平台合规）
5. ~~大理冷启动~~（替换为：**冷启动选具体小群体，不通吃**）
6. **教育只说一次 · 不抢夺注意力**
7. **不为焦虑的人增加心理负担**
8. **真实 > 看起来繁荣**

### 4 条 vibe coding 时代独有的延伸原则

**§9 反 AI 装大佬**（哲学 2 的本产品落地）
- 截图 + AI 文案 ≠ 作品
- 真实性核验靠**可访问产物**（陌生人能自己玩到），不靠文案表演

**§10 反技术中心化**（§6.6 的本产品落地）
- 弱信息群体（不懂代码的创造者）是首要服务对象
- 强信息群体（懂代码的）作为兼容群体，不是反过来
- UI 语言：Notion / Linear / Figma 美学，**不是** GitHub 技术美学

**§11 反 AI 焦虑营销**（哲学 7 的本产品落地）
- 主动屏蔽 / 降权的内容形态："X 天月入 $Y"、"AI 替代你"、"必学 N 个工具"
- 写进社区规则
- "在做但没做完"的人有空间，不只是 winner 舞台

**§12 多问 AI 和自己 · 反一锤定音**（元原则 · 应用前 11 条时的协作姿势）

vibe coding 时代的产品决策不该一锤定音 —— spec v0.1 不是终点，AI 的判断不是终点，本能反应也不是终点。**每次"按字面" 或 "按反射" 决定的瞬间，都要被反复拷问一次**。

三个反复要问的问题：

1. **字面禁止 = spec 真想禁止吗？**
   - 例：工作室 mini-stats 字面违反 "不显示累计"，但**自己看**是里程碑感（反 AI 焦虑营销的抗体），不是社交货币 → 改成 isMe 守卫，访客看不到、自己保留
2. **字面允许 = spec 真想允许吗？**
   - 例：spec §5.2 写"可按工具标签筛选"，但工具只是辅助 · 重要的是想法 · 按工具分类正是 GitHub 技术语言 → 砍掉工具 filter（违反 §10 精神）
3. **这是不是一个还不存在的问题？**
   - 例：alpha 阶段 feed 每天几条到几十条，filter / toggle 解决的是"信噪比下降"问题 · 现在根本不存在这个问题 → 砍掉 chip filter + mode toggle

**几个观察的结果**：
- 来回拷问的产物经常是**减法** —— 砍功能比加功能需要更多勇气
- AI 的反射动作是"spec 写了就照做" + "现有产品有 X 就 X"，需要用户用真实使用经验拷问
- 用户的反射动作是"我看到不满意就改"，需要 AI 用 spec 精神 + 第一性原理拷问
- 拷问出来的设计经常是 spec v0.1 当时没想到的更对的形态（spec 是活文档，应该被拷问后更新）

**典型反模式**：
- AI 给出"spec 第 X 节明确禁止"就关闭讨论
- 用户说"我觉得这样不对"就照改、不追问"违反哪条哲学 / 是真违反还是字面违反"
- 任一方说服另一方，但没有共识 spec 的精神
- **按片段读代码 / 按 grep 命中下结论 · 没看完整上下文**（context 残缺反模式）
  - 应用三问**之前**必须读完整上下文：CSS 改造历史不只是命中那行、函数 if-guard 要完整看、spec 段落看周边几条、commit log 看历史演化
  - 反例 1：v0.81 我 grep 到 `actions-sql.js:444` 的 `notifType = willStuck ? 'projectStuck' : 'ownerUpdate'` 就推断"每条 update 都通知关注者 = spam"，实际 line 432 已经有 `if (notifyTinkered || willStuck)` 守卫 + checkbox 默认不勾 = opt-in
  - 反例 2：about-container 我 grep 到历史注释「560px 中文最佳行宽」就推断 about 比其他页面窄，实际 v0.66.1 早就改成 `width: 100%` 跟 main 同宽
  - 这种反模式拷问出来的是**虚假痛点**：要么已被解决、要么实际不存在。三问的产物会很有迷惑性（看起来逻辑通顺）但跟现实脱节

## 四、核心需求（马斯洛锚定 · 不可砍）

| # | 需求 | 马斯洛层 | 满足方式 |
|---|---|---|---|
| **#1** | **给在做但没做完的人留主舞台** | 归属 + 心理安全 | 工作室持续状态 · 项目时间线 · "在做"是 first-class |
| **#2** | **真实复用反馈链** | 尊重 + 自我超越 | "接走 / 跑通了"动作 · 反馈是真实复用，不是点赞 |
| **#3** | **作品必须挂可访问产物** | 心理安全（#1 #2 的前置） | 项目必填产物链接 · 反"AI 装大佬" |

**这三条互相支撑，少任何一条另外两条立不住。**

```
   #3 (产物锚定真实)
        ↓
   保证 #1 (在做的人不被假装做完的挤压)
        ↓
   保证 #2 (跑通反馈有真实可核验的标的)
        ↓
   反过来让 #1 的"在做"有了希望路径
```

## 五、产品形态

### 5.1 核心实体

#### 工作室（Workshop）

用户的主体 —— 不是"账号 + 帖子列表"，是一个对外开放的工作现场。

**包含**：
- 工作室名（可笔名 / ID / 真名 · **不强制实名**）
- 一句话「我在做的事」
- 项目陈列架（三层：最近在动 / 也在做 / archive · 按 commit 时间自动分层）

**不包含**（显性约束）：
- ❌ 粉丝数 / 关注数
- ❌ 作品总数
- ❌ JOINED N days
- ❌ bio / 自我介绍（只有「我在做什么」）

#### 项目（Project）

产品最重要的实体。

**必填字段**：
| 字段 | 说明 |
|---|---|
| 项目名 | |
| 一句话描述 | |
| **可访问产物链接** | v0 / Bolt / Lovable / Replit / Claude Artifact / 自定义 GPT / GitHub / 自部署 URL 等 |
| 当前状态 | 🟢 进行中 / 🟡 卡住 / ⏸ 暂停 / ✅ 跑通了 / 📦 归档 |

**可选字段**：
- 「用的 AI 工具」标签（多选 · 如 `via Cursor + Claude` `via v0` `via Bolt + Supabase`）
- 时间线（进展事件）
- 互动区（便签 / 反馈 / 接走）

**项目数量**：
- 工作室项目数**不限**
- 自动按 commit 时间分层
- 30 天没动 → 自动沉到"也在做"
- **让真实活动决定哪些被看见，不需要用户主动整理**

#### 进展（Update）

> ⚠️ 内部叫 commit · UI 上**绝不用 commit 这个词**

每条进展是项目时间线的一条记录：
- 「跑通了 X」
- 「卡在 Y」
- 「试了 Z 失败」
- 可附图、可附提示词（带复制按钮 —— 这是 #2 真实复用的入口）

#### 反馈动作（3 级）

| 等级 | 动作 | 门槛 | 系统记录 |
|---|---|---|---|
| 👀 | 看了 | 隐形 | 仅记录浏览（不展示）|
| 🛠 | 想试试 | 一键 | "进入想试试队列" |
| 🤝 | 做了延伸版 | 必须挂自己项目 | "张三基于这个做了 X" |

> **2026-06-05 修订**：原 spec 设 4 级反馈，但实测 "✨ 觉得有意思" 跟"想试试"的语义区别太弱 · 且容易退化为隐形点赞 · 砍掉这一档。保留的 3 档：浏览（隐性）/ 想试试 / 延伸版，各有清晰的行为承诺梯度。

**关键约束**：
- 显示具体是谁，**不显示总数**
- 首页**绝不按此排序**
- 创作者主页**不显示累计**
- 「想试试」可升级为「做了延伸版」—— **两阶段渐进承诺**

#### 便签（评论替代）

不是评论 —— **不能嵌套盖楼**。

- 每条便签独立平等
- 时间倒序到底
- 创作者要回，直接 commit 一条进展（不是回复评论）

#### 方法（Method）/ 方法库

进展（update）升格而来 · 别人能 borrow 复用的一份手艺（方法 / 踩坑 / 上手指南 / 决策推演）。

**两层 · 读者不一样**（2026-06-13 想清楚的）：
- **卡片层** —— 给人扫的。一句话 + 标签 + 被借次数 · 人在这一层决定「要不要复制」。
- **正文层（markdown）** —— 给 AI 用的。复制进自己的 AI 对话里当上下文 · 所以可以长、可以结构化。

**卡片那句话（scenario）的文风** —— 这句是卡片的成败点 · 必须直给：
- 站在「借的人」视角 · 说清「借了它能解决什么问题 / 得到什么能力」· 不是介绍这段在讲什么
- 动词或问题开头讲收益 · 短 · 不超过约 25 字 · 让人一眼判断要不要复制
- 不讲故事不讲过程 · 不用内部代号黑话
- 反例（别这样）：把正文开头的故事钩子顶上来 ——「桥卡 7 小时 · 一个测试占位暗号引发的静默失踪案」
- 正例：「AI 协作一端突然收不到消息？先查测试占位暗号」

**生成 / 填充**：
- `tinker contribute --auto` 时 LLM 按上面文风自动生成（见 cli `llmPickSections` 的 prompt · 2026-06-13 已改成问「借了能解决什么」）
- 也可 `tinker edit-method <id> --scenario "..."` 人手写 / 改
- scenario 是这句的唯一来源 · **卡片不该在 scenario 空时回退去显示正文**（当前 webapp 有这个回退 · 待修 · 跟 docs/10-tech-debt.md ③ 那批一起做）

### 5.2 关键界面

#### 主屏（不是 feed，是「今天有动静的工作室」）

- 卡片来源 = **项目事件**（commit / 卡住 / 跑通 / 开工作室）
- 时间倒序 · 无算法 · 无热门 · 无 trending
- 卡片高度一致（不做大小图差异，反"急"视觉强化）
- 空 feed **不刷数据**
- 可按工具标签筛选

#### 工作室页（用户的 home）

- 头部：名字 + 「我在做的事」
- 中部：项目陈列架（三层）
- 底部：最近动静时间线

#### 项目页（产品最重要的页面）

- 头部：名字 + 描述 + 产物链接 + 工具标签 + 状态
- 中部：时间线（每条 commit 平等 · 无点赞排序）
- 底部：反馈区（3 级反馈 + 便签墙 + 接走过的人）

#### 入场仪式（新用户开工作室前）

**不是 onboarding wizard · 是「先做一个 5 分钟超小作品」**

```
新用户点「开我的工作室」
        ↓
"开张之前, 先做一个 5 分钟的小东西"
        ↓
3 个 starter 题目让 ta 挑 (例如):
  · 用 Claude Artifact 做一个倒计时
  · 用 v0 做一个咖啡选择器
  · 用 Bolt 做一个待办列表
(每个都附完整提示词 + 工具链接)
        ↓
做完了, 挂上产物链接
        ↓
工作室开张, 第一个项目自动登记
```

**关键约束**：
- starter 题目**定期换**（不变成考试模板）
- 不允许跳过（已有作品 → 直接挂上来也算 starter 完成）
- **没有 Lv1 / Lv2 等级**

### 5.3 核心动作链

#### 浏览者路径

```
打开 App → "今天有动静"首页
       ↓
点进某个项目 → 看 ta 在做什么 / 卡在哪
       ↓
三种反馈 (按代价从轻到重):
  · 留便签 (10 秒)
  · 想试试 (一键)
  · 做了延伸版 (挂自己项目)
       ↓
创作者收到的不是 "+1 赞"
而是「李四接走了你的项目做了求职信生成器」
```

#### 创作者路径

```
开工作室 → 强制走 5 分钟入场仪式
       ↓
挂第一个项目 (强制有产物)
       ↓
有进展就 commit 一条
       ↓
卡住时 commit「卡在 X」
       ↓ 系统自动通知:接走过你项目的人 + 关注者
       ↓
跑通时 commit「跑通了」
       ↓ 通知接走过的人:"你接走的项目跑通了"
```

#### 离开路径（不挡用户离开）

- 每个项目页有「一键分享到 X / 小红书 / 微信」
- **离开社区也行，但产物链接永远在**
- 对应 petfinder 第 1 条「不在乎用户在哪聊」

## 八、冷启动策略

### 第一天的画面
- 开发者本人 + 5-10 个种子用户开各自的工作室
- 每人挂 1-2 个真在做的项目
- 主屏显示真实 commit / 卡住 / 跑通
- **不刷数据** —— "这里只有 8 个人在做事"也比假活跃强

### 产品自己作为第一个工作室 ✨

**"Tinker 这个产品本身"就是第一个公开工作室。**

- 每个开发 commit 都变成一条公开进展
- 解决"第一天空 feed"问题 —— 用真实开发占满第一个月
- 同时是**最好的营销资产**：一个用 vibe coding 做的、给 vibe coder 的工作室，每步开发都公开

### 第一个陌生用户路径

1. 直接进主屏（无 onboarding · 无注册墙）
2. 看到真实工作室真实进展
3. 可匿名浏览全部
4. 想互动时才需要开自己的工作室
5. 开工作室时必走入场仪式 → 5 分钟做一个超小作品

## 九、评估新功能的判断问题（继承 petfinder · 本产品新增第 9 个）

任何新功能上之前必问一遍。**任一项答案不理想 → 默认不做**：

1. **工程**：长期维护成本？
2. **法律**：合规红线？
3. **情感**：让用户更焦虑还是更安心？
4. **真实**：让数据更真实还是更"看起来繁荣"？
5. **运营**：上线后谁来持续维护？
6. **行为可执行性**（§6.5）：用户那个时刻真的会做这个操作吗？
7. **视角对称性**（§6.6）：不同用户群体的能力 / 信息差异是什么？
8. **情感时刻不数据化**（§6.7）：这个功能把善意 / 事件商品化吗？
9. **反技术中心化**（本产品新增 §10）：这个功能 / 字段 / 词汇假设用户懂代码吗？不懂代码的人能用吗？
10. **反一锤定音**（本产品新增 §12）：这个判断是按 spec 字面 / 按反射决定的，还是被来回拷问过 spec 精神？字面禁止的 = 真禁止吗？字面允许的 = 真允许吗？这个问题真的存在吗？
